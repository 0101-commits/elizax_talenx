## F1. 목표 진단 이원화 — "잘 쓴 목표" vs "잘 굴러가는 목표"

### 현행
- QW7 리포트(`js/tx_agent.js:638-694`)는 이미 ①문장 품질(중복·미연계·측정불가) ②운영 신호(체크인 공백·진척 정체) 2섹션으로 분리 렌더. 설계 주석(`:673`)에 분리 의도 명시.
- 운영 신호는 실데이터 계산(`qw7OpsRows()` `:609-637`): checkins 최근일 gap, progress<30 정체.
- QW1(`:697-731`)은 주간 체크인 선제 감지 팝업으로 운영 신호 전용 채널이 별도 존재.

### 갭
분리는 "한 리포트 안의 2섹션"에 머묾. 받는 팀장 입장에서는 **타이밍·처방·행동 단위가 안 갈라짐**:
- 처방 버튼이 섹션 하단 공통 버튼 1벌 — 행 단위 액션 없음.
- 품질 진단은 수립 마감 전에 봐야 의미 있고, 운영 진단은 상시(주간)여야 하는데 둘 다 "요청 시 1회 리포트".

### 기획안
1. **채널 분리**: 품질 진단 → "목표수립 마감 D-3 게이트 리포트"(1회성, 수정 제안 중심). 운영 진단 → QW1 주간 신호에 통합(상시, 리마인드·1:1 중심). QW7은 두 채널의 통합 조망으로 유지.
2. **행 단위 처방 버튼**: 품질 행 = [수정 제안 보내기]·[병합 제안]·[elizax로 정제], 운영 행 = [체크인 리마인드]·[1:1 제안]. 모두 기존 승인 게이트 경유.
3. **진단 출처 라벨**: 각 행에 "수립 품질 | 운영 신호" 배지 — 팀장이 "고칠 문제 vs 챙길 문제"를 즉시 구분.
4. 운영 신호 임계값 명세화: 체크인 gap ≥14일 warn / ≥30일 bad, 진척 드리프트(기간경과율−진행률 ≥20%p) 신규 추가.

### 시장 근거
Betterworks: 궤도 10% 이탈 시 Slack 자동 알림. Viva Goals: 팀별 체크인 주기 설정+nudge 버튼. Profit.co: 실시간 health score. → "품질=수립 게이트, 건강도=드리프트%·경과일"이 확립된 관행.

**난이도 하 · 기존 QW1/QW7 재배선 중심**

## F2. 측정불가 KR 사전 차단 — 린트 확장 + 저장 게이트 ★사용자 지목 quick win

### 현행
- 인라인 린트 바(`js/tx_upgrade.js:173-248`): goal 스코프 4규칙(업계Top·구축완료·향상으로 끝남·노력표현) 이미 구현. "업계 Top 수준", "체계 구축 완료" 정확히 커버.
- 오프라인 집계: `scripts/enrich_talenx_data.py:221` target_value 숫자 유무 → `kr_measurable_rate`.

### 갭
- 린트가 **문장 패턴만** 봄. KR의 구조 필드(target_value·mode·기준선)와 교차 검증 없음.
- 경고가 떠도 **그대로 저장 가능** — 게이트 없음. "쓸 때 잡는다"는 취지 미완.
- 팀장 QW7의 "측정 불가" 행과 인라인 린트가 **별개 하드코딩** — 같은 규칙 엔진 공유 안 함.

### 기획안
1. **구조 린트 3규칙 추가** (기존 goal-1~4에 얹기):
   - goal-5: target_value에 숫자 없음 → bad "목표 수치가 없습니다"
   - goal-6: 기준선(현재값) 미입력 → warn "무엇에서 출발하는지 없으면 달성률을 다툽니다"
   - goal-7: mode="여부(binary)"인데 검증 조건 텍스트 없음 → warn "무엇이 되면 완료인지 판정 조건 필수"
2. **저장 게이트**: bad 칩 존재 상태로 저장 시 확인 모달 — "이 표현은 평가 시점에 측정할 수 없습니다" + [elizax로 정제]/[그대로 저장(사유 입력)]. 사유는 원장에 `type:goal` 기록 → 평가 시점에 "알고도 남긴 근거"가 남음.
3. **규칙 엔진 단일화**: LINT_RULES를 `window.EZLint`로 승격, 인라인 바·QW7 품질 섹션·수립 게이트 리포트(F1)가 동일 규칙 ID 공유. QW7 측정불가 행에 규칙 ID·해당 문구 하이라이트 표시.
4. AI 정제 프롬프트에 mode·target_value 컨텍스트 포함(현재는 문장만 전송).

### 시장 근거
Lattice/Leapsome은 AI 제안 수준, 규칙 기반 measurability checker는 **어느 제품도 공식화 안 함** — 공백. KR 측정 타입(#·$·%·binary) 강제는 Lattice 관행 → mode 필드와 교차 검증이 자연스러운 확장.

**난이도 하 · POC 검증 완료 항목 · 1순위 권장**

## F3. 난이도 표기에 근거 필수화

### 현행
- 편집 UI에 난이도 select(S/A/B) + 근거 input(`txf-krdiffwhy`, placeholder "전년 실적 대비 +30% 상향")까지 이미 존재(`js/tx_fix_perf.js:749-756`). 안내문도 정확히 취지 반영.
- 데이터 스키마에 구조화 필드 존재: `difficulty_basis:{type:"peer"|"yoy", label, note}`.
- **그러나** ①UI 입력이 저장 미연결(토스트만) ②평가 점수식에 난이도 완전 미반영(evaluations에 계수 없음) ③캘리브레이션의 "난이도 보정계수 반영"(`tx_agent.js:963`)은 표시 문구뿐.

### 기획안
1. **근거 입력 구조화**: 자유 텍스트 → [비교 대상 select: 전년 실적(yoy)/동료·직군 평균(peer)/시장·고객 기준(market)] + [수치/설명 입력] 2필드. `difficulty_basis` 스키마 그대로 매핑. market 타입 신설.
2. **S 난이도는 근거 필수**: 근거 없이 S 선택 시 저장 게이트(F2와 동일 모달 패턴). B는 선택.
3. **평가 화면 노출**: 평가 입력 카드(tx_fix_appr)에서 KR 달성률 옆에 배지 "난이도 S · 전년 +30% 상향" — 평가자가 "80% 달성이지만 어려운 목표였다"를 근거와 함께 판단. **핵심은 점수 자동 보정이 아니라 판단 재료 노출**(자동 보정은 캘리브레이션 논쟁 대상이라 데모에선 표시까지만).
4. **캘리브레이션 연결**: QW 캘리브레이션 화면의 "난이도 보정" 문구를 실데이터 참조로 — 난이도 분포(팀별 S 비율)와 근거 유무율 표시. "근거 없는 S = 캘리브레이션 리스크" 행 추가.

### 시장 근거
구글 계보 committed(1.0 기대)/aspirational(0.7 기대) 이원 태깅이 표준 — 같은 0.63이 유형에 따라 실패/준수로 뒤집힘. 단 "난이도 판단 근거를 수립 시점에 구조화 기록"하는 전용 필드는 주요 제품에 없음 — **공백**. committed 미달 시 post-mortem 의무가 유일한 유사 장치.

**난이도 하~중 · F2와 같은 저장 게이트 패턴 재사용**

## F4. 상태 저장소 — 평가 근거는 지워지지 않는다 (W1 리뷰 1순위)

### 현행
- localStorage `elizax_ctx_v1:<emp_id>`, **80건 초과 시 오래된 것부터 삭제**(`js/tx_ctx_ledger.js:54,145-148`).
- 레코드에 `usedCount`(답변 인용 횟수)·`weight`(기여도 1-3) 필드 이미 존재.
- 정책 모달(`tx_policy.js:104-106`)이 스스로 "실서비스에선 이러면 안 됨" 고지.

### 기획안 — 2층 구조: 보존 정책 정의(문서) + 데모 구현(코드)

**(a) 보존 정책 정의** — 3등급:
| 등급 | 대상 | 보존 | 근거 |
|---|---|---|---|
| 불변 보존 | 평가 근거로 인용된 기록(usedCount>0), 확정 평가·등급, 승인 게이트 결정 | 사이클 종료 후 **3년**, 수정 불가(정정은 append) | 근로기준법 §42(승급·감급 서류 3년), 분쟁 제기 시 파기 금지(EEOC 관행 동일) |
| 사이클 보존 | 체크인·1on1 확정 요약·피드백 | 당해+직전 사이클 | 차기 목표수립 재료(F6) |
| 롤링 | 화면 진입 등 임시 맥락 | 80건 롤링 유지 | 현행 그대로 |

**(b) 데모 구현**:
1. **핀 고정(pinned)**: `usedCount>0 || weight===3 || type==="eval"` 레코드는 80건 제한 대상에서 제외. `saveStore()` 삭제 루프에서 pinned 분리 — 코드 몇 줄.
2. **보존 배지**: 원장 패널 항목에 "평가 인용됨 · 보존 대상" 배지 + 정책 모달의 "최근 80건" 문구를 3등급 표로 교체.
3. **변조 탐지 데모**: 불변 등급 레코드에 `prev_hash` 체이닝(간단 문자열 해시) — 원장 패널에 "체인 검증 ✓" 표시. write-once 스토리를 눈으로 시연. (SuccessFactors Form Audit Trail·WORM 관행 대응)
4. **내보내기/가져오기**: 원장 JSON export/import 버튼 — "브라우저를 떠나도 산다" 시연 + 데모 데이터 백업 실용성.
5. 서버측 저장소(CF Worker KV)는 **다음 단계로 명시만** — 이번 범위 아님.

**난이도 중 · 1~2는 소규모, 3~4는 반나절**

## F5. 민감정보 노출 규칙 — 매트릭스를 표시에서 강제로

### 현행
- 역할×기록 노출 매트릭스(full/summ/anon/no)가 `js/tx_policy.js:62-75`에 **정적 표시(설명 모달)로만** 존재 — 실제 마스킹 로직 없음.
- 실제 작동하는 게이팅은 근거 깊이(core/trace/logic, `tx_entry.js:66` + `tx_ctx_ledger.js:537-542`)뿐 — 이건 "얼마나 깊이"이지 "봐도 되는가"가 아님.
- 상향 피드백 행이 매트릭스에서 하이라이트(팀장=익명·집계만) — 취지는 이미 선언됨.

### 기획안
1. **정책 엔진화**: MATRIX를 선언형 함수로 승격 — `EZPolicy.check(viewerRole, recordType, relation) → "full"|"summ"|"anon"|"no"`. 데이터는 기존 tx_policy 테이블 재사용(새 규칙 발명 없음, 표시=강제 단일 소스).
2. **3개 소비 지점에 적용**:
   - 원장 패널·근거 스트립: `no`→항목 자체 미표시, `anon`→"익명 집계 (N명)" 카드로 치환, `summ`→요약문만.
   - **AI 도구 게이트**: `tx_ai_tools.js` 도구 응답에 동일 정책 적용 — 팀장이 elizax에게 "나에 대한 상향 피드백 원문 보여줘" 해도 익명 집계만 반환. **"화면을 우회해도 AI가 같은 규칙을 지킨다"가 이 기능의 데모 하이라이트.**
   - 차단 시 침묵이 아니라 명시: "응답자 보호 정책에 따라 익명 집계로 제공합니다 (정책 v3.1)" + 정책 모달 링크.
3. **익명 임계값 N=3**: 상향 피드백 응답자 3명 미만이면 집계조차 비공개 — "응답 인원이 적어 결과를 표시하지 않습니다". mock 데이터에 2명 응답 팀 1개 심어 시연.
4. 차순위 소그룹 숨김(Lattice strong protection: 차집합 역산 차단)은 정책 모달에 소개만.

### 시장 근거
Lattice 익명 임계값 최소 3(하향 조정 불가), Culture Amp 통상 5. Culture Amp는 360을 "익명 아님"으로 공지하는 게 오히려 표준 — 임계값+명시 고지 조합이 업계 골격.

**난이도 중 · 정책 데이터는 있고 배선만 — AI 게이트가 차별화 서사**
